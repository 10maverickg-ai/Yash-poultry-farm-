Yash Poultry Farm — Database Schema Specification
(Phase 1)

Context for whoever builds this (including Claude Code)
This schema digitizes an existing paper-based record-keeping system for a layer poultry
farm (Yash: ~1,30,000 total capacity — 1,00,000 layer + 30,000 chick/grower combined,
typically ~98,000 layers filled; 8+ sheds, BV300 breed, in-house feed mill). The design
principle: the schema mirrors the paper registers exactly — it does not introduce new
fields the farm doesn’t already track, except where explicitly noted as a deliberate addition.

Capture method: supervisors photograph paper registers (their existing habit). OCR extracts
data into these tables. A human reviewer (the owner) sees only flagged/exception records —
not a full confirm screen for every entry.

Two deliberate additions beyond pure digitization (owner’s explicit choice):

1. Daily egg size-grading counts (not previously written down)

2. Flock Register as a formal table (previously “in supervisors’ and owner’s heads”)




Table 0: farms (reference table)
Two farms under the same owner, physically adjacent but operationally separate —
separate registers, separate supervisors, no shared production data. One narrow exception:
eggs occasionally move between Yash and Anil, in either direction, when a buyer’s
collection car arrives and one farm is short on trays to fill the order — trays are pulled from
the other farm to complete that specific sale. This is sale-driven and rare, not a routine
transfer, and is captured as a signed adjustment (+/-) with a note in daily_egg_stock ,
matching how it’s already written on the register — reconciliation is inferred from the sign
and note rather than a strict linked-record structure. Beyond this, resources like labour, raw
material, or machinery are occasionally shared but not yet mapped from paper — out of
scope for v1.

Table 4: daily_egg_stock
One row per day, whole farm, evening. Captures the reconciliation ledger + grading, pooled
across all sheds (grading is not flock-specific, confirmed).


Design: header + flexible ledger (not fixed columns)
The register doesn’t use fixed categories every day — it’s a running ledger: start with a total,
then a variable list of labeled +/- entries (sale, breakage, gift, cross-farm adjustment, etc.),
ending in a closing balance that carries forward as tomorrow’s opening figure. Rather than
forcing this into fixed columns (which breaks the first day a new entry type shows up),
model it as a header row + child ledger table. OCR/AI reads whatever label is written (e.g.
“Anil(-)”, “Sale(-)”, “Birth(+)”), categorizes it, and the running balance is calculated
automatically — this is explicitly the reconciliation logic the owner wants AI to own, not fixed
data entry.

Table 5: sales
Not a separate paper register — there is no standalone sales book. Sales are represented
by entries in the daily_egg_stock_entries ledger itself: QC-passing eggs go to Farm Fresh
(main channel), reject/small-size eggs go to wholesalers/distributors (e.g. “Sukha,” a
recurring buyer). This table exists to hold sale-specific detail (buyer, price, amount) that the
ledger entry alone doesn’t carry, and is populated from ledger entries where category =
sale , not extracted independently.

Price is confirmed to have no paper source at all — it’s set daily by market demand/supply
and never written down. price_per_unit and amount are therefore optional fields with no
OCR extraction target in v1; they stay null unless the owner chooses to enter them manually
later (e.g. reconciling against a periodic Farm Fresh statement, if that ever becomes
available).

What’s explicitly OUT of v1
   Weekly Health — no paper record exists currently. Not built in v1. Medicine/vaccine
   logging flagged as a strong v2 candidate once the core system is running (likely already
   happening informally).

   Per-flock feed formulation tracking in daily_production — not possible since
   formulation is set by group, not individually validated per bag.

   Automated ingredient-consumption-to-bags-produced cross-check — no fixed
   ratio exists (confirmed), so this validation rule is not implemented.
