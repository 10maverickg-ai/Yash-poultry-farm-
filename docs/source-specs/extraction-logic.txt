Yash Poultry Farm — OCR Extraction Logic Spec (Phase 3)
Companion to yash-poultry-farm-schema-spec.md . This document defines how photographed
register pages get read and mapped into the database tables, and how the flagging/review-queue
logic works in practice.

Build sequencing reminder: this is Phase 3. Phase 1 (schema) and Phase 2 (manual entry screens,
owner-facing) should exist and hold real data before this phase starts — OCR extraction needs a
working schema to test against, and manual entry gives you a way to validate the schema
independently of extraction accuracy.




General extraction pipeline (applies to every register type)
1. Photo intake — supervisor/manager uploads photo via phone. Store raw photo
   ( source_photo_url ) regardless of what happens downstream — this is the audit trail and what
   the owner reviews against when a flag fires.

2. Page-type detection — identify which register this is (Daily Production / Egg Stock Ledger /
   Sales / Feed Stock / Feed Formulation) from layout/header cues, since a supervisor might
   photograph any of these on a given day.

3. Field extraction — OCR reads each field per the register-specific mapping below. Every
   extracted field gets a confidence score.

4. Structural/arithmetic validation — run the validation rules already defined in the schema spec
   (HD% check, ledger balance check, feed balance check, etc.) against the extracted values,
   independent of OCR confidence. This is the step that catches a confidently-misread number, not
   just an illegible one (e.g. the quintal/kg case from schema design).

5. Farm/date/flock resolution — resolve farm_code , date , and (for Production)
    flock_internal_id via flock_label_history .

6. Flag or commit — if all checks pass, write directly to the database. If any check fails (see flag
   triggers below), the record is written with flagged = true and routed to the owner’s review
   queue, alongside the photo and the specific field(s) in question.

7. Owner review — owner sees only flagged records, photo side-by-side with extracted value,
   corrects the field directly. Correction writes to the record and clears the flag.




Flag triggers (apply across all register types)
A record gets flagged if any of the following are true:

   Low OCR confidence on any individual field (illegible/ambiguous handwriting)

   Missing field — expected column blank in the photo (bad lighting, fold, skipped entry)

   Arithmetic/structural mismatch — extracted values don’t satisfy the register’s own internal
   math (see schema spec’s validation rules per table)

   Outlier/range check — e.g. mortality >3x that flock’s trailing 7-day average, HD% outside
   plausible range for flock age — flagged even if OCR was fully confident, since this may be a real
   event worth the owner’s eyes, not necessarily an OCR error

   Cross-table mismatch — e.g. Egg Stock ledger’s total_eggs_produced doesn’t match the sum
   of that day’s Daily Production entries

   Low-confidence category classification — for Egg Stock ledger lines, when AI can’t confidently
   classify a label into sale/breakage/gift/etc.

   No confident sale match — ledger line categorized as sale but no matching Sales record found
   within a reasonable date/amount window

Everything that passes all checks writes straight through — no owner review needed. This keeps the
review queue small and meaningful rather than a full confirm-everything screen.

6. Flock Register events (placement, stage transitions, renumbering)
Confirmed as low-frequency, largely manual events rather than a daily OCR target. Recommend
building this as a simple manual-entry form (owner or manager triggered) rather than an OCR
pipeline — the volume doesn’t justify extraction complexity, and getting renumbering right matters
more than getting it automated (see schema spec’s internal-ID/display-label design).

Hatchery bill photos are the one OCR-relevant input here — extract source_hatchery , date, and
chick count when a new flock is placed.

Additional flag trigger: tray-scale vs egg-scale magnitude check

Numbers in these registers can represent either TRAYS or EGGS depending on
magnitude (1 tray = 30 eggs). Use magnitude as a disambiguation heuristic
during extraction:

  - Tray-scale: up to roughly a few thousand.
  - Egg-scale: tens of thousands (e.g. "28482" is an egg count, never trays).

This is NOT a new field — it is a sanity check on extracted values: if an
extracted number's magnitude does not fit the unit scale expected for that
column (tray-scale where trays are expected, egg-scale where eggs are
expected), FLAG the record for owner review rather than accepting the value
at face value — same family as the existing outlier/range checks.

Implication for extraction: all stored egg quantities in the schema are in
EGGS (tray figures are display-only, computed /30). A tray-scale number
found where an egg count is expected (or vice versa) suggests the paper
recorded the other unit that day. Do NOT silently auto-convert on magnitude
alone — magnitude is a signal, not proof. Flag it; the owner confirms the
unit in review, exactly as with the quintal/kg check in Feed Stock.
